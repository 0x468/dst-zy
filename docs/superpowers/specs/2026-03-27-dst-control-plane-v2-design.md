# DST 控制平面 V2 设计

## 背景

当前仓库的 V1 已经解决了 DST runtime 镜像本身的核心问题：

- DST dedicated server 镜像可以稳定构建与运行
- 支持 `install-only` / `update` / `validate` / `never`
- 支持 server mod 的 `runtime` / `prewarm` / `skip`
- 已有 Cluster 的迁移路径已经清楚
- 用户文档、迁移文档、验证记录与预检脚本已经基本对齐

但 V1 的定位仍然是“运行时镜像 + 文档 + 验证”，并不提供真正的控制平面。对于日常管理多个 DST 集群，当前仍存在明显缺口：

- 缺少统一的集群列表和状态面板
- 缺少导入已有 Cluster 的可视化流程
- 缺少表单化配置编辑
- 缺少统一的启动、停止、重启、更新、校验入口
- 缺少操作任务记录和审计轨迹
- 高级用户虽然可以直接改文件，但普通使用路径仍然偏底层

因此需要一个新的 V2 子项目，作为现有 DST runtime 的“控制平面（control plane）”。

## 目标

- 提供一个可在单机部署、面向少数可信用户的 Web 管理面板
- 一个面板实例管理多个 DST 集群条目
- 支持导入已有 Cluster、创建新 Cluster、编辑配置、启停更新和基础日志查看
- 配置编辑以表单化为主，同时保留高级模式编辑原始文件
- 保持底层配置与运行结果可见、可导出、可手工接管，不做黑盒系统
- 保持资源占用、部署复杂度和安全面在第一阶段可控

## 非目标

- 不做多租户 SaaS
- 不做复杂 RBAC 与细粒度权限矩阵
- 不做远程 agent 或多宿主机管理
- 不做 Kubernetes 或更重的容器编排系统
- 不做第三方 OAuth / SSO 作为第一阶段必需项
- 不做在线终端、任意 shell 执行或任意宿主机文件浏览
- 不把管理面板与 DST runtime 混成一个镜像

## 用户与使用场景

第一阶段用户画像明确为：

- 你自己
- 少数可信用户
- 同一台宿主机上的集群管理员

典型场景：

1. 导入一个已有 Cluster 并纳入管理
2. 创建一个新的 DST 集群并生成基础配置
3. 通过表单修改常用配置项
4. 在高级模式下直接修改 `ini` / `lua` / compose 文件
5. 启动、停止、重启、更新、校验某个集群
6. 查看最近任务、错误与基础运行状态
7. 做简单备份或导出集群目录

## 方案比较

### 方案 1：轻量后台型

- 单个后端服务直接渲染页面
- 少量 JavaScript 增强交互
- 底层围绕文件和 `docker compose`

优点：

- 开发快
- 部署简单
- 资源占用低

缺点：

- 面板交互上限较低
- 多集群状态、高级配置编辑、日志和任务体验会较快变差
- 后续扩展复杂交互时会逐步吃力

### 方案 2：全动态控制台型

- 前后端分离
- 后端直接以 Docker API 为主编排
- 面板数据库作为主要状态来源

优点：

- 产品感最强
- 运行时控制最灵活

缺点：

- 开发复杂度高
- 状态分散在数据库、Docker 和文件系统之间
- 排障和手工接管成本高
- 第一阶段对单机可信用户场景来说过重

### 方案 3：混合控制平面型

- 前后端分离
- 后端维护管理元数据
- 真实配置落地为文件
- 执行层继续基于 `docker compose`
- 高级模式直接查看和编辑原始文件

优点：

- 兼顾可用性和可维护性
- 面板不是黑盒，文件和 compose 仍然可见
- 迁移、导出、备份、排障都相对直接
- 比直接以 Docker API 做主编排更适合第一阶段

缺点：

- 需要处理“管理元数据”和“真实文件状态”的同步
- 架构比纯后端渲染稍复杂

## 选型

选择方案 3。

原因：

- 第一阶段核心诉求是“可管理、可维护、可导出”，而不是“极致自动化”
- 单机、可信用户场景下，`docker compose` 仍然是足够稳妥的执行边界
- 后续如果需要更强编排能力，也可以在保留文件落地的前提下逐步演进

## 总体架构

V2 不是对现有 DST runtime 的直接替换，而是一个独立项目，和 runtime 保持组合关系：

- 现有 DST runtime 镜像：继续负责单个集群的运行逻辑
- 新的控制平面项目：负责管理多个集群的元数据、配置和执行操作

第一阶段部署拓扑：

- 同一台宿主机
- 一个控制平面服务
- 多个由控制平面生成并管理的 DST 集群容器

控制平面负责三类职责：

1. 管理元数据
2. 文件落地与配置解析
3. 执行编排与任务审计

## 模块拆分

### 1. `web-ui`

前端控制台，负责：

- 登录页
- 集群列表页
- 集群详情页
- 表单化配置页
- 高级模式编辑页
- 任务记录与基础日志页

### 2. `api-server`

Go 后端 API，负责：

- 鉴权与会话
- 集群元数据 CRUD
- 配置读取、校验与保存
- 状态聚合
- 任务调度入口

### 3. `runtime-manager`

执行层，负责：

- 调用 `docker compose`
- 获取容器状态
- 执行启动、停止、重启、更新、校验
- 收集任务执行输出

### 4. `cluster-files`

文件层，负责：

- 解析和回写 `cluster.ini`
- 解析和回写两个 shard 的 `server.ini`
- 管理 `dedicated_server_mods_setup.lua`
- 管理 `modoverrides.lua`
- 生成集群级 compose / env 文件
- 管理受控目录结构

### 5. `audit-jobs`

任务与审计层，负责：

- 记录用户操作
- 记录任务状态
- 记录错误输出和时间线
- 为后续的日志页和审计页提供数据

## 数据模型

第一阶段数据库只存“管理元数据”和“缓存快照”，不把数据库变成唯一真相源。

### `ClusterRecord`

字段建议：

- `id`
- `slug`
- `display_name`
- `note`
- `cluster_name`
- `base_dir`
- `compose_file`
- `env_file`
- `status`
- `created_at`
- `updated_at`

职责：

- 表示面板中的一个集群条目
- 保存显示名、备注、路径与当前聚合状态

### `ClusterPorts`

字段建议：

- `cluster_id`
- `master_host_port`
- `caves_host_port`
- `steam_host_port`
- `caves_steam_host_port`
- `master_server_port`
- `caves_server_port`
- `master_master_server_port`
- `caves_master_server_port`

职责：

- 保存宿主机端口和解析得到的 shard 内部端口
- 供表单回显与冲突检查使用

### `ClusterConfigSnapshot`

字段建议：

- `cluster_id`
- `snapshot_version`
- `cluster_ini_json`
- `master_server_ini_json`
- `caves_server_ini_json`
- `updated_at`

职责：

- 保存最近一次成功解析后的结构化配置快照
- 供前端表单直接消费

### `JobRecord`

字段建议：

- `id`
- `cluster_id`
- `job_type`
- `status`
- `requested_by`
- `stdout_excerpt`
- `stderr_excerpt`
- `started_at`
- `finished_at`

职责：

- 表示一次集群操作任务
- 为任务页、日志页和错误排障提供基础数据

### `AuditRecord`

字段建议：

- `id`
- `actor`
- `action`
- `target_type`
- `target_id`
- `summary`
- `created_at`

职责：

- 记录“谁在什么时候做了什么”
- 保留最基本的管理追踪能力

## 单一真相源策略

这是 V2 成败的关键设计点。

约定如下：

- 真实运行配置以文件为准
- 数据库只保存：
  - 管理元数据
  - 解析后的结构化快照
  - 任务和审计记录

这意味着：

- 表单保存时，最终必须回写真实文件
- 高级模式编辑原文后，后端必须重新解析并刷新结构化快照
- 如果快照和文件冲突，以文件重新解析结果为准

这样做的好处是：

- 用户仍然可以离开面板手工维护
- 集群目录可以直接导出、备份或迁移
- 面板损坏时不会把真实配置锁死在数据库里

## 部署拓扑

第一阶段保持单机部署，不做远程 agent。

逻辑上的运行组件：

- `dst-control-plane-api`
- `dst-control-plane-web` 构建产物
- 多个 DST 集群容器

数据库优先选择 SQLite，本地文件化存储，不额外引入 PostgreSQL 或 Redis。

## 宿主机目录布局

建议由控制平面统一管理一个根目录，例如：

```text
/opt/dst-control-plane/
  app/
  data/
    app.db
    clusters/
      cluster-a/
        meta/
        compose/
        runtime/
          steam-state/
          dst/
          ugc/
          data/
    backups/
    logs/
```

职责划分：

- `app.db`
  控制平面数据库
- `clusters/<slug>/meta/`
  面板缓存元数据与快照
- `clusters/<slug>/compose/`
  此集群的 compose / env 文件
- `clusters/<slug>/runtime/`
  实际挂给 DST 容器的运行目录
- `backups/`
  备份产物
- `logs/`
  控制平面自身运行日志和任务日志

## 功能范围

### 第一阶段必须有

- 集群列表
- 新建集群
- 导入已有集群
- 表单化配置编辑
- 高级模式原文编辑
- 启动、停止、重启、更新、校验
- 基础日志与任务记录
- 基础备份入口
- 基础审计记录

### 第一阶段明确不做

- 多租户和复杂 RBAC
- 跨机器管理
- 在线终端
- 任意文件浏览
- 实时高吞吐日志系统
- 自动扩缩容
- 自动漏洞修复平台

## 配置编辑策略

### 表单模式

第一阶段表单覆盖高频字段：

- 集群名称、描述、密码、游戏模式、人数
- shard 端口与宿主机端口
- `cluster_key`
- `DST_UPDATE_MODE`
- `DST_SERVER_MODS_UPDATE_MODE`

### 高级模式

第一阶段允许直接编辑：

- `cluster.ini`
- `Master/server.ini`
- `Caves/server.ini`
- `dedicated_server_mods_setup.lua`
- `modoverrides.lua`
- 生成出的 compose / env 文件

高级模式必须带：

- 明确风险提示
- 保存前校验
- 覆盖写入前确认

## 执行编排策略

第一阶段不直接以 Docker API 作为主编排，而是继续使用 `docker compose`：

- 新建集群时，生成此集群的 compose 文件
- 启动时执行 `docker compose up -d`
- 停止时执行 `docker compose stop`
- 重启时执行 `docker compose restart`
- 更新/校验时通过 env 切换 `DST_UPDATE_MODE`

原因：

- 更贴近当前仓库已有能力
- 用户容易理解
- 生成结果透明，易于手工接管
- 对单机场景足够稳妥

## 安全边界

第一阶段虽然面向少数可信用户，但仍然必须具备基础安全能力。

### 必须有

- 面板自身登录能力
- 密码哈希存储
- 安全会话管理
- 所有写操作鉴权
- 输入校验和路径约束
- 危险操作确认
- 审计日志

### 部署建议

- 默认监听本机或内网地址
- 公网部署时放在 Caddy/Nginx/Zero Trust 后面
- 即使在反代后面，面板自身仍保留基础登录

### 明确禁止

- 前端直接接触 Docker socket
- 任意 shell 命令执行
- 用户输入任意宿主机路径并直接操作
- 在不受控目录外读写文件

## 技术选型

### 后端

- 语言：Go
- 路由：标准库或 `chi`
- 数据层：轻量 SQL 层，避免第一阶段引入过重 ORM
- 数据库：SQLite

### 前端

- 语言：TypeScript
- 框架：React
- 构建：常规现代前端构建工具即可

### 为什么不是 Rust

Rust 后端并非不可行，但对这个项目第一阶段来说：

- 真正的难点不在极限性能
- 而在配置模型、文件系统编排、任务流与交互体验
- Go 在这种控制平面场景下能更快、更稳地交付第一阶段

## 资源占用目标

第一阶段目标不是极致低资源，而是“足够轻且结构清楚”：

- 一个 Go 后端服务
- 一个前端构建产物目录
- 一个 SQLite 文件
- 不引入 Redis、PostgreSQL、消息队列

这意味着控制平面本身应明显轻于若干实际运行的 DST 集群容器。

## 风险与约束

### 风险 1：表单快照和原始文件不同步

应对：

- 以文件为准
- 每次高级编辑后强制重新解析
- 保存失败时明确阻止状态更新

### 风险 2：执行层过度开放

应对：

- 所有执行动作模板化
- 路径白名单
- 严禁直接把任意命令拼给 shell

### 风险 3：范围膨胀

应对：

- 第一阶段只做单机、可信用户场景
- 不提前做远程 agent、多租户和重型运维能力

## 验证策略

在实现阶段至少需要覆盖：

- 后端配置解析与回写测试
- 路径约束与输入校验测试
- 执行层任务编排测试
- API 基础鉴权测试
- 前端关键交互与表单流程测试
- 至少一条端到端流程：
  - 创建集群
  - 保存配置
  - 启动
  - 停止

## 第一阶段里程碑建议

### 里程碑 1：后端骨架与数据模型

- 后端服务启动
- SQLite 初始化
- 登录与会话基础能力
- 集群元数据 CRUD

### 里程碑 2：文件层与导入/创建流程

- 生成受控目录结构
- 创建新集群
- 导入已有集群
- 基础配置解析与快照

### 里程碑 3：执行层与生命周期管理

- 启动、停止、重启
- 更新、校验
- 任务记录与错误输出

### 里程碑 4：前端控制台

- 集群列表
- 详情页
- 表单配置页
- 高级模式页

### 里程碑 5：审计、备份与收尾

- 审计页
- 基础备份入口
- 部署文档和安全建议

## 结论

V2 应该被视为“新的控制平面项目”，而不是在当前 runtime 仓库上随手叠几个管理脚本。

第一阶段最佳路线是：

- 单机
- 少数可信用户
- 前后端分离
- Go 后端 + TypeScript 前端
- 文件为真相源
- `docker compose` 为执行边界

这样既能显著提升多集群管理体验，又不会在第一阶段把系统做成难以维护的黑盒。
