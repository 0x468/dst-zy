# 控制平面安全说明

## 威胁模型

第一阶段控制平面的目标不是“直接裸露到公网”。它的默认威胁模型是：

- 同一台主机
- 自己使用，或少数可信用户
- 可以接受单管理员账号
- 可以在控制平面前面再加一层反向代理或零信任访问

如果你的目标是开放给不特定用户，当前版本还不够。

## 已实现的安全边界

### 1. 受控根目录

控制平面只会在 `DST_CONTROL_PLANE_DATA_ROOT` 之下创建和管理集群。导入已有集群时，也要求导入源位于这个根目录内。

这意味着：

- 不会允许随便读取 `/home`、`/etc` 等宿主机路径
- slug 中的 `../`、反斜杠等路径穿越写法会被拒绝
- 控制平面只对自己管理的目录负责

### 2. 文件是真相源

数据库只保存管理元数据，不保存完整游戏配置。

这意味着：

- 即使控制平面数据库损坏，真实 Cluster 文件仍然可单独恢复
- 你可以脱离控制平面直接检查 `cluster.ini`、`server.ini`、mods 配置
- 安全审计时可以直接对比落盘文件，不必猜数据库状态

### 3. 会话签名

当前登录态使用服务端签名的会话 cookie。你必须自行设置安全的 `DST_CONTROL_PLANE_SESSION_SECRET`，不要保留仓库默认值。

至少要做到：

- 生产环境换成随机高熵密钥
- 密钥不要提交进 Git
- 更换密钥前预期旧会话会失效

### 4. 基础登录失败限流

当前版本已经为 `/api/login` 增加最小限流能力，主要用于降低单机场景下的低成本爆破风险。

默认值是：

- `5` 次失败尝试
- `5` 分钟时间窗

对应环境变量：

- `DST_CONTROL_PLANE_LOGIN_RATE_LIMIT_MAX_ATTEMPTS`
- `DST_CONTROL_PLANE_LOGIN_RATE_LIMIT_WINDOW`

注意：

- 这是第一阶段的进程内限流，不是分布式限流
- 服务重启后计数会重置
- 它只能算基础缓解，不等于公网级防护

## 当前未覆盖的部分

下面这些能力还没有完整实现，不能误以为“已经有了”：

- 多用户与 RBAC
- 细粒度 API 授权
- 2FA / WebAuthn
- CSRF 专项防护策略梳理
- 公网场景下的审计告警与封禁策略

因此，当前版本不应直接裸露在公网。

## 部署建议

如果要远程访问，推荐最小做法：

1. 控制平面服务只监听内网地址，或只绑定到回环地址。
2. 前面放一层 Caddy、Nginx、Traefik 之类的反向代理。
3. 反代层加 TLS。
4. 最好再叠加一层可信身份入口，例如 VPN、Tailscale、Cloudflare Zero Trust 或内网穿透后的访问控制。

控制平面自身的登录，只能算“应用层第一道门”，不能代替网络边界。

## 凭据与敏感文件

下面这些都应视为敏感信息：

- `DST_CONTROL_PLANE_ADMIN_PASSWORD`
- `DST_CONTROL_PLANE_SESSION_SECRET`
- Cluster 内的 `cluster_token.txt`
- 可能包含私有服配置的原始 ini 文件

建议：

- 全部通过环境变量、`.env` 或秘密管理系统注入
- 不要提交到仓库
- 备份时确认是否需要对这些文件单独加密

## Docker 执行边界

第一阶段运行控制仍然走 `docker compose`，这既是实现边界，也是安全边界。

优点：

- 控制平面不需要直接长期持有更复杂的 Docker API 编排能力
- 每个集群的执行配置落在自己的 compose 文件里，更容易审计

代价：

- 运行控制能力受 compose 文件表达范围限制
- 如果宿主机 Docker 权限本身过大，控制平面具备的影响面也会跟着扩大

所以，真正的最小权限设计还需要后续继续收敛。

如果你使用单镜像部署 compose，还会额外挂载 `/var/run/docker.sock`。这意味着：

- 控制平面容器具备很强的宿主机 Docker 控制能力
- 不适合不可信多用户环境
- 更适合单机、自用、可信管理员场景

## 日志与审计

当前已经有基础任务记录和审计表，但仍属于第一阶段能力：

- 适合追踪谁触发了什么动作
- 适合看最近一次任务是否成功
- 不足以替代完整安全审计平台

如果要进入更严肃的使用场景，后续至少还要补：

- 更明确的审计事件模型
- 失败登录与敏感配置修改日志
- 导出与保留策略
- 与反向代理访问日志的关联
